Why Reproducibility Matters

A 2016 Nature survey found 70% of researchers failed to reproduce others' experiments. Computational work should be the easiest to reproduce — but only if you follow good practices. The goal: anyone should be able to re-run your entire analysis from raw data to final figures with a single command.

The Reproducibility Stack

Level 1: Code in version control (Git)
Level 2: Environment captured (Conda / Docker)
Level 3: Pipeline automated (Snakemake / Nextflow)
Level 4: Data archived (SRA / Zenodo / checksums)
Level 5: Documentation complete (README, methods)

Snakemake — Python-Based Workflows

Snakemake is the most popular workflow manager in bioinformatics. It uses Python syntax, tracks file dependencies, and only re-runs steps whose inputs changed.

Basic Snakefile

# Snakefile
SAMPLES = ["sample1", "sample2", "sample3"]

rule all:
    input:
        expand("results/counts/{sample}_counts.txt", sample=SAMPLES),
        "results/multiqc_report.html"

rule fastqc:
    input: "data/raw/{sample}_R1.fastq.gz"
    output: "results/qc/{sample}_R1_fastqc.html"
    threads: 2
    shell: "fastqc {input} -o results/qc/ -t {threads}"

rule trim:
    input:
        r1="data/raw/{sample}_R1.fastq.gz",
        r2="data/raw/{sample}_R2.fastq.gz"
    output:
        r1="results/trimmed/{sample}_R1.fastq.gz",
        r2="results/trimmed/{sample}_R2.fastq.gz",
        html="results/trimmed/{sample}_fastp.html"
    threads: 4
    shell:
        "fastp -i {input.r1} -I {input.r2} "
        "-o {output.r1} -O {output.r2} "
        "--html {output.html} --thread {threads} "
        "--qualified_quality_phred 20 --length_required 50"

rule align:
    input:
        r1="results/trimmed/{sample}_R1.fastq.gz",
        r2="results/trimmed/{sample}_R2.fastq.gz",
        idx="data/reference/genome.fa"
    output: "results/aligned/{sample}.bam"
    threads: 8
    shell:
        "bwa mem -t {threads} {input.idx} {input.r1} {input.r2} | "
        "samtools sort -@ 4 -o {output} && "
        "samtools index {output}"

rule count:
    input:
        bam="results/aligned/{sample}.bam",
        gtf="data/reference/genes.gtf"
    output: "results/counts/{sample}_counts.txt"
    threads: 4
    shell:
        "featureCounts -a {input.gtf} -o {output} -T {threads} "
        "-p --countReadPairs {input.bam}"

rule multiqc:
    input: expand("results/qc/{sample}_R1_fastqc.html", sample=SAMPLES)
    output: "results/multiqc_report.html"
    shell: "multiqc results/ -o results/ --force"

Running Snakemake

# Dry run — see what would execute
snakemake -n

# Run with 8 cores
snakemake --cores 8

# Generate workflow DAG visualization
snakemake --dag | dot -Tpng > workflow.png

# Run on a cluster (SLURM)
snakemake --cores 100 --cluster "sbatch -p normal -n {threads} -t 4:00:00"

# Use conda environments per rule
snakemake --cores 8 --use-conda

# Force re-run of a specific rule
snakemake --cores 8 --forcerun align

Conda Integration

# In Snakefile, specify per-rule environments:
rule align:
    input: ...
    output: ...
    conda: "envs/alignment.yml"   # Snakemake creates this env automatically
    shell: "bwa mem ..."

# envs/alignment.yml
channels:
  - bioconda
  - conda-forge
dependencies:
  - bwa=0.7.17
  - samtools=1.17

Nextflow

Nextflow is another popular workflow manager, especially in large-scale production pipelines (nf-core).

// main.nf
params.reads = "data/raw/*_{R1,R2}.fastq.gz"
params.ref = "data/reference/genome.fa"
params.outdir = "results"

process TRIM {
    publishDir "${params.outdir}/trimmed"
    input: tuple val(sample), path(reads)
    output: tuple val(sample), path("${sample}_trimmed_{1,2}.fastq.gz")
    script:
    """
    fastp -i ${reads[0]} -I ${reads[1]} \
          -o ${sample}_trimmed_1.fastq.gz -O ${sample}_trimmed_2.fastq.gz \
          --qualified_quality_phred 20
    """
}

process ALIGN {
    publishDir "${params.outdir}/aligned"
    input: tuple val(sample), path(reads)
    output: tuple val(sample), path("${sample}.bam")
    script:
    """
    bwa mem -t ${task.cpus} ${params.ref} ${reads} | samtools sort -o ${sample}.bam
    samtools index ${sample}.bam
    """
}

workflow {
    reads_ch = Channel.fromFilePairs(params.reads)
    trimmed = TRIM(reads_ch)
    ALIGN(trimmed)
}
# Run Nextflow
nextflow run main.nf --reads 'data/*_{R1,R2}.fastq.gz' -profile docker

# nf-core: community-curated production pipelines
nextflow run nf-core/rnaseq --input samplesheet.csv --genome GRCh38 -profile docker
Snakemake vs Nextflow

Snakemake: Python-based, easier to learn, great for custom pipelines, strong in academia.
Nextflow: Groovy-based, better cloud/container support, nf-core has production-ready pipelines, strong in industry.
Both are excellent. Pick one and learn it well.

Containers — Freeze Your Environment

Conda captures package versions, but containers capture the entire operating system. This is the gold standard for reproducibility.

Docker

# Dockerfile
FROM continuumio/miniconda3:latest

# Install tools
RUN conda install -y -c bioconda -c conda-forge \
    samtools=1.17 bwa=0.7.17 fastqc=0.12.1 \
    star=2.7.11a subread=2.0.6 \
    snakemake=7.32 multiqc=1.17

WORKDIR /workspace
COPY Snakefile .
COPY envs/ envs/

CMD ["snakemake", "--cores", "all"]
# Build
docker build -t rnaseq-pipeline:v1.0 .

# Run (mount data directory)
docker run -v $(pwd)/data:/workspace/data \
           -v $(pwd)/results:/workspace/results \
           rnaseq-pipeline:v1.0

Singularity (for HPC clusters)

# Most HPC clusters don't allow Docker (security). Use Singularity instead:
singularity pull docker://biocontainers/samtools:v1.17
singularity exec samtools_v1.17.sif samtools --version

# Run a container
singularity exec samtools_v1.17.sif samtools sort input.bam -o sorted.bam

Data Management with SQLite

For large result sets (millions of variants, expression values), SQLite provides fast querying without a server.

import sqlite3

# Create database
conn = sqlite3.connect("results/variants.db")
conn.execute("""
    CREATE TABLE IF NOT EXISTS variants (
        chrom TEXT, pos INTEGER, ref TEXT, alt TEXT,
        qual REAL, filter TEXT, genotype TEXT, gene TEXT, impact TEXT
    )
""")

# Load variants from VCF
with open("annotated.vcf") as f:
    for line in f:
        if line.startswith("#"): continue
        fields = line.strip().split("\t")
        conn.execute(
            "INSERT INTO variants (chrom,pos,ref,alt,qual,filter) VALUES (?,?,?,?,?,?)",
            (fields[0], int(fields[1]), fields[3], fields[4], float(fields[5]), fields[6])
        )
conn.commit()

# Query: high-quality variants on chromosome 1
for row in conn.execute(
    "SELECT chrom, pos, ref, alt, qual FROM variants WHERE qual > 50 AND chrom='chr1' LIMIT 10"
):
    print(row)

# Aggregate: count variants per chromosome
for row in conn.execute(
    "SELECT chrom, COUNT(*) as n FROM variants GROUP BY chrom ORDER BY n DESC"
):
    print(f"{row[0]}: {row[1]} variants")

The Complete Reproducibility Checklist

Before Publishing Your Analysis
  • ☐ All code in Git with meaningful commit messages
  • ☐ Environment captured: conda environment.yml AND/OR Dockerfile
  • ☐ Pipeline automated: Snakemake/Nextflow/Makefile
  • ☐ Raw data archived with accession numbers (SRA, GEO)
  • ☐ Checksums for all downloaded files
  • ☐ README documents every step, tool version, and parameter
  • ☐ All figures generated by scripts (not manually edited)
  • ☐ Software versions recorded in methods section
  • ☐ Can be re-run from scratch with a single command
  • ☐ Code and data deposited in a public repository (GitHub + Zenodo)
