Chapter 5: Version Control with Git

~60 min Beginner Git, GitHub, Branches, Collaboration

Why Git?

Imagine this scenario: you modify your analysis script, the results change, and you can't figure out what you changed or go back to the working version. Git prevents this by tracking every change you make, letting you:

  • Go back in time — Restore any previous version of any file
  • Track what changed and why — Every change has a message explaining it
  • Collaborate safely — Multiple people can work on the same project without overwriting each other
  • Experiment freely — Create branches to try new approaches without breaking the main code

Git's Mental Model

Working Directory → Staging Area → Repository (commits)
     (edit files)     (git add)      (git commit)

Think of it like:
  1. You edit files on your desk (working directory)
  2. You put finished work in an outbox (staging area / git add)
  3. You file the outbox into the cabinet (repository / git commit)

Getting Started

# Configure Git (once per machine)
git config --global user.name "Your Name"
git config --global user.email "you@email.com"
git config --global core.editor "nano"       # Or vim, code, etc.
git config --global init.defaultBranch main  # Use 'main' not 'master'

# Initialize a new repository
cd my_project
git init

# Or clone an existing one from GitHub
git clone https://github.com/username/project.git
cd project

The Daily Workflow

# 1. Check status — what's changed?
git status

# 2. See the actual changes
git diff                    # Unstaged changes
git diff --staged           # Staged changes (ready to commit)

# 3. Stage changes
git add script.sh           # Stage one file
git add scripts/            # Stage entire directory
git add -A                  # Stage everything (new, modified, deleted)

# 4. Commit with a descriptive message
git commit -m "Add trimming step with fastp, min quality Q20"

# 5. View history
git log --oneline           # Compact view
git log --oneline --graph   # Visual branch history
git log -5                  # Last 5 commits
Good Commit Messages

Bad: "update", "fix", "changes"
Good: "Add fastp trimming with Q20 threshold and 50bp min length"
Good: "Fix off-by-one error in BED coordinate conversion"
Your future self will thank you.

.gitignore for Bioinformatics

Large data files and generated outputs should NOT be tracked in Git. Create a .gitignore file:

# .gitignore for bioinformatics projects

# Raw and processed data (too large for Git)

*.fasta
*.fa
*.bam
*.bai
*.sam
*.vcf
*.vcf.gz
*.bed

# Results (regenerated by pipeline)
results/

# Reference genomes
*.fa.fai
*.dict
*.bt2
*.amb
*.ann
*.bwt
*.pac
*.sa

# R artifacts
.Rhistory
.RData
.Rproj.user/

# Python artifacts

# Snakemake
.snakemake/

# OS files

Thumbs.db
What TO track in Git

Scripts (.sh, .R, .py), configuration files (Snakefile, Makefile, environment.yml), documentation (README.md, notes), and small metadata files (sample sheets, gene lists). NOT large data or binary files.

Branching — Safe Experimentation

Branches let you try new approaches without affecting the main code:

# Create and switch to a new branch
git checkout -b new-normalization

# ... make changes, commit them ...
git add scripts/normalize.R
git commit -m "Try TMM normalization instead of DESeq2 default"

# Switch back to main
git checkout main
# main is unchanged — your experiment is safely on the branch

# If the experiment worked, merge it in
git merge new-normalization

# If it didn't work, just delete the branch
git branch -d new-normalization

Collaborating via GitHub

# Connect local repo to GitHub
git remote add origin https://github.com/username/project.git

# Push your commits to GitHub
git push -u origin main     # First time (sets upstream)
git push                    # Subsequent pushes

# Get your collaborator's changes
git pull                    # Fetch + merge

# If there's a conflict:
# 1. Git marks the conflicting sections in the file
# 2. You manually edit to resolve
# 3. git add the resolved file
# 4. git commit

Undoing Things

# Unstage a file (undo git add)
git reset HEAD file.txt

# Discard changes to a file (revert to last commit)
git checkout -- file.txt

# Amend the last commit message
git commit --amend -m "Better message"

# Go back to a specific commit (view only)
git checkout abc1234

# Revert a commit (creates a new commit that undoes it)
git revert abc1234

# Stash changes temporarily (save work without committing)
git stash
# ... do other work ...
git stash pop               # Restore stashed changes

Practical Example: Bioinformatics Git Workflow

# Day 1: Start project
mkdir rnaseq_project && cd rnaseq_project
git init
mkdir -p {data/{raw,processed},results,scripts,envs}
echo "# RNA-Seq Analysis" > README.md
cp ~/templates/gitignore .gitignore
git add -A
git commit -m "Initialize project structure"

# Day 2: Add QC script
nano scripts/01_qc.sh
git add scripts/01_qc.sh
git commit -m "Add FastQC quality control script"

# Day 3: Add alignment, push to GitHub
nano scripts/02_align.sh
conda env export > envs/environment.yml
git add scripts/02_align.sh envs/environment.yml
git commit -m "Add BWA alignment script and conda environment"
git push

# Day 5: Try a different aligner on a branch
git checkout -b try-star
nano scripts/02_align_star.sh
git add scripts/02_align_star.sh
git commit -m "Test STAR aligner as alternative to BWA"
# ... compare results ...
git checkout main
git merge try-star
git push
