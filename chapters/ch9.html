Chapter 9: Sequence Formats & QC

~75 min Intermediate FASTA, FASTQ, Phred, FastQC, Trimming

FASTA Format

The simplest and most universal sequence format. Used for reference genomes, protein sequences, assembled contigs, and more.

>seq1 Homo sapiens BRCA1 mRNA
ATGGATTTTATCTGCTCTTCGCGTTGAAGAAGTACAAAATGTCATTAATGCTATGCAGAA
AATCTTAGAGTGTCCCATCTGTAAGTGGAGAAAGCTTCTGTGAATGATATTCCTGTGCTT
>seq2 Mus musculus TP53
ATGACTGCCATGGAGGAGTCACAGTCGGATATCAGCCTCGAGCTCCCTCTGAGCCAGGAG

Rules:

  • Header line starts with > followed by ID and optional description
  • Sequence can span multiple lines (typically 60-80 characters per line)
  • No quality information — just the sequence
  • Can contain nucleotides (ATCGN) or amino acids (ACDEFGHIKLMNPQRSTVWY)

IUPAC Ambiguity Codes

N = any base (A, C, G, or T)
R = purine (A or G)
Y = pyrimidine (C or T)
S = strong (G or C)
W = weak (A or T)
K = keto (G or T)
M = amino (A or C)

FASTQ Format

The standard output of sequencing machines. Like FASTA but with quality scores for every base.

@SRR1234567.1 HWI-ST999:100:C0JHTACXX:1:1101:1235:2026 1:N:0:ATCACG
ATCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCG
+
IIIIIIIIIHIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII

Four lines per read, always:

  1. @ — Header with read ID and instrument info
  2. Sequence — The actual bases (ATCGN)
  3. + — Separator (sometimes repeats the header)
  4. Quality — One ASCII character per base, encoding error probability
FASTQ ≠ 4 × number of reads in wc -l

While FASTQ normally has 4 lines per read, quality strings can contain @ characters. Never use grep -c "^@" to count FASTQ reads — it can overcount. Use echo $(($(wc -l < file.fastq) / 4)) or dedicated tools.

Phred Quality Scores — Understanding Base Quality

Each character in the quality string encodes the probability that the base call is wrong:

Q = -10 × log₁₀(P_error)

Score  Error Rate   Accuracy   Meaning
─────  ──────────   ────────   ───────
Q10    1 in 10      90.0%      Poor
Q20    1 in 100     99.0%      Acceptable
Q30    1 in 1,000   99.9%      Good (typical threshold)
Q40    1 in 10,000  99.99%     Excellent

ASCII Encoding (Phred+33, Illumina 1.8+)

Quality character → ASCII value → subtract 33 → Phred score

Character:  !  "  #  $  %  &  ...  5  ...  ?  ...  I  ...
ASCII:      33 34 35 36 37 38     53     63     73
Q score:    0  1  2  3  4  5      20     30     40

Example quality string: IIIIIHHHFF
Scores:                 40 40 40 40 40 39 39 39 37 37

Quality Control with FastQC

# Run FastQC on a single file
fastqc reads.fastq -o results/qc/

# Run on multiple files with 4 threads
fastqc data/raw/*.fastq -o results/qc/ -t 4

# Aggregate all FastQC reports into one summary
multiqc results/qc/ -o results/multiqc/

Key FastQC Metrics

  • Per Base Sequence Quality — Should be >Q20 across most positions. Quality typically drops at the 3' end of reads
  • Per Sequence Quality Scores — Most reads should have average quality >Q30
  • Per Base Sequence Content — Should be roughly equal A≈T and G≈C. Bias at the start is normal (random hexamer priming)
  • GC Content — Should match expected distribution for your organism (~40% for human). A second peak suggests contamination
  • Sequence Duplication — High duplication may indicate PCR amplification bias or low library complexity
  • Adapter Content — Should be minimal. High adapter content means reads are longer than the DNA fragments
  • Overrepresented Sequences — May indicate adapter contamination or rRNA contamination
Don't Panic About FastQC Warnings

FastQC was designed for whole-genome sequencing. RNA-Seq data will often show "failures" for GC content and duplication — this is expected because highly expressed genes produce many identical reads. Context matters.

Trimming — Cleaning Your Reads

fastp (Recommended — fast, modern, all-in-one)

fastp \
  -i reads_R1.fastq.gz -I reads_R2.fastq.gz \
  -o trimmed_R1.fastq.gz -O trimmed_R2.fastq.gz \
  --qualified_quality_phred 20 \
  --length_required 50 \
  --detect_adapter_for_pe \
  --thread 4 \
  --html results/fastp_report.html \
  --json results/fastp_report.json

Trimmomatic

trimmomatic PE \
  reads_R1.fastq.gz reads_R2.fastq.gz \
  trimmed_R1.fastq.gz unpaired_R1.fastq.gz \
  trimmed_R2.fastq.gz unpaired_R2.fastq.gz \
  ILLUMINACLIP:TruSeq3-PE.fa:2:30:10 \
  LEADING:3 TRAILING:3 \
  SLIDINGWINDOW:4:20 \
  MINLEN:50

Trimmomatic steps explained:

  • ILLUMINACLIP — Remove adapter sequences
  • LEADING:3 — Remove leading bases below quality 3
  • TRAILING:3 — Remove trailing bases below quality 3
  • SLIDINGWINDOW:4:20 — Scan with 4-base window, cut when average drops below Q20
  • MINLEN:50 — Drop reads shorter than 50 bp after trimming

Practical Workflow: QC → Trim → Re-QC

#!/bin/bash
set -euo pipefail

SAMPLE=$1
mkdir -p results/{qc_raw,qc_trimmed,trimmed}

# Step 1: QC raw reads
fastqc data/raw/${SAMPLE}_R{1,2}.fastq.gz -o results/qc_raw/

# Step 2: Trim
fastp -i data/raw/${SAMPLE}_R1.fastq.gz -I data/raw/${SAMPLE}_R2.fastq.gz \
      -o results/trimmed/${SAMPLE}_R1.fastq.gz -O results/trimmed/${SAMPLE}_R2.fastq.gz \
      --qualified_quality_phred 20 --length_required 50 \
      --html results/trimmed/${SAMPLE}_fastp.html

# Step 3: QC trimmed reads (verify improvement)
fastqc results/trimmed/${SAMPLE}_R{1,2}.fastq.gz -o results/qc_trimmed/

echo "Done! Compare results/qc_raw/ vs results/qc_trimmed/"
